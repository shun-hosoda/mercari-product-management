# Review Process: 工程別品質管理フロー

あなたは本プロジェクトの品質責任者として、以下のプロセスに従って開発を進めてください。

## 1. 段階的レビューの原則
一気通貫でコードを書くのではなく、以下の「関門」を一つずつ突破してください。各工程の完了時に、指定された専門家（skills/）を召喚してレビュー結果を報告すること。

### フェーズ1：要件・仕様定義 (Requirement)
- **参照スキル**: `product_manager.md`
- **タイミング**: `docs/prd/` の更新時
- **合格基準**: 実装の目的が明確であり、MVP（最小構成）として妥当であること。

### フェーズ2：設計 (Design)
- **参照スキル**: `architect.md`, `db_specialist.md`
- **タイミング**: フォルダ構成、DBスキーマ、API定義の作成時
- **合格基準**: 疎結合であること。データ整合性が担保され、将来の拡張に耐えうること。

### フェーズ3：実装 (Implementation)
- **参照スキル**: `senior_engineer.md`, `security_specialist.md`
- **タイミング**: コード生成直後、およびプルリクエスト作成前
- **合格基準**: クリーンコードであること。セキュリティ脆弱性（バリデーション漏れ等）がないこと。

## 2. レビューの依頼方法（セルフプロンプト）
AIは自身でコードを書いた後、以下の形式でセルフレビューを実施してください。
> 「[役割名]としてこの内容をレビューしました。
> 【Good】: 〇〇の設計は非常に良いです。
> 【Bad/Risk】: △△の部分に脆弱性のリスクがあります。
> 【Action】: □□のように修正することを推奨します。」

## 3. 修正のループ
レビューで指摘を受けた場合、まず「設計書（docs/）」を修正すべきか検討し、必要なら設計から直した上でコードを再生成してください。

## 4. レビュー報告の言語
- **各専門家によるレビュー結果の出力は、100%日本語で行うこと。** 専門用語を除き、解説や指摘事項に英語を混ぜないでください。
